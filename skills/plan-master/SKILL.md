# Plan Master - 智能计划管理

通过多轮对话收集用户需求，生成结构化的个人计划，支持计划的全生命周期管理。

## 核心流程

```
步骤1：用户信息收集
├─ 用户提出计划诉求
├─ 模型询问更多信息（如需）
└─ 用户回答信息

步骤2：生成计划并保存
├─ 基于收集的信息生成计划
├─ 冲突检测
├─ 保存到 Markdown 文件
└─ ⚠️ 展示计划详情 + 小气泡

步骤3：计划管理
├─ 查看/修改/删除计划
├─ 生成日程提醒（用户主动发起）
└─ 生成 HTML 报告
```

---

## ⚠️ 强制要求

### 1. 回复内容与小气泡必须同时存在

> **回复内容和小气泡是独立的两个输出，必须同时存在，缺一不可！**
>
> - **回复内容**：向用户展示的详细信息（计划详情、操作结果等）
> - **小气泡**：通过 `RENDER_MINI_CARD` 工具渲染的快捷操作按钮
>
> **两者互不冲突，互不影响，必须同时输出！**

**正确做法**：
```
✅ 步骤1：输出详细的回复内容（包含完整计划详情）
✅ 步骤2：调用 RENDER_MINI_CARD 工具渲染小气泡
✅ 两者都执行，用户既能看到详情，又能点击快捷按钮
```

**错误做法**：
```
❌ 只输出简短回复 + 小气泡（缺少计划详情）
❌ 只输出计划详情，不调用小气泡
❌ 认为两者冲突，二选一
```

### 2. 计划详情必须完整展示

> **创建、修改、查看计划时，回复内容必须包含完整的计划详情！**
>
> 计划详情包括：标题、类型、状态、目标、时间周期、任务清单、内容安排、冲突提示（如有）

**展示示例**：

### 📅 行程安排
| 日期 | 时段 | 活动 | 交通 | 用餐 | 提示 |
| 5月1日 | 上午 | 斗兽场 | 步行 | 附近餐厅 | 提前预约 |

### 3. 小气泡必须渲染

> **计划管理阶段的所有回复，在输出内容后，必须调用 `RENDER_MINI_CARD` 工具！**

| 场景 | 是否需要小气泡 |
|:---|:---:|
| 询问用户更多信息 | ❌ 不需要 |
| 计划创建后 | ✅ 必须 |
| 计划修改后 | ✅ 必须 |
| 删除计划后 | ✅ 必须 |
| 删除事项后 | ✅ 必须 |
| 查看计划列表 | ✅ 必须 |
| 查看计划详情 | ✅ 必须 |
| 生成日程后 | ✅ 必须 |
| 生成报告后 | ✅ 必须 |

### 4. 小气泡调用方式

> **必须通过 `RENDER_MINI_CARD` 工具实现，禁止在回复文案中写任何小气泡文字！**

```json
{
  "cardType": "actions",
  "cardData": {
    "actions": [
      { "text": "生成网页报告", "link": "sendchat://为这个计划生成一份网页报告" }
    ]
  }
}
```

### 5. 禁止自动创建日程

> **日程创建必须由用户主动发起！**
>
> 计划保存后，不能自动帮用户创建日程。只有用户明确说"生成日程""创建提醒"时才执行。

## 步骤1：用户信息收集

### 1.1 识别用户意图

用户提出计划诉求后，判断信息是否足够生成计划。

**信息充足**：包含计划类型 + 核心要素（时间、目标、约束等）

**信息不足**：缺少关键信息，需要提问

### 1.2 提问收集信息

根据缺失的关键信息提问：

| 计划类型 | 核心信息 |
|:---|:---|
| travel | 目的地、时间、天数、人数、预算、偏好 |
| fitness | 目标、周期、当前状况、可用时间 |
| study | 目标、考试时间、当前水平、可用时间 |
| work | 项目目标、截止时间、团队情况 |
| life | 目标、周期、具体事项 |

**提问原则**：
- 每次提问 2-3 个问题
- 用自然对话语气
- 用户已明确说的信息不再确认
- 提问时**不需要**渲染小气泡

### 1.3 识别计划类型

| 类型 | 关键词 | 内容表格格式 |
|:---|:---|:---|
| travel | 旅游、旅行、出行、行程 | 时段表（日期、时段、活动、交通、用餐、提示）|
| fitness | 健身、减肥、运动、减脂 | 周表（周次、周一~周日）|
| study | 学习、备考、考试、读书 | 阶段表（阶段、时间、内容、目标、资料）|
| work | 项目、工作、OKR、任务 | 里程碑表（里程碑、时间、交付物、负责人）|
| life | 生活、日常、习惯、家庭 | 日表（日期、任务、备注）|

## 步骤2：生成计划并保存

### 2.1 生成计划

基于收集的用户信息，生成结构化计划。

**输出格式**（Markdown + YAML Front Matter）：

```markdown
---
id: plan-{uuid}
type: travel|fitness|study|work|life
title: 计划标题
status: active
period:
  start: YYYY-MM-DD
  end: YYYY-MM-DD
goal: 核心目标
createdAt: ISO-8601
updatedAt: ISO-8601
---

# 计划标题

## 任务清单

| 事项 | 截止 | 优先级 |
|:---|:---|:---:|
| 事项1 | YYYY-MM-DD | high |
| 事项2 | YYYY-MM-DD | medium |

## 内容安排

（根据计划类型使用对应表格格式）
```

**事项优先级**：

| 优先级代码 | 用户展示 | 含义 |
|:---|:---|:---|
| `high` | 🔴 高 | 关键事项，建议设置日程提醒 |
| `medium` | 🟡 中 | 重要事项，建议记录关注 |
| `low` | 🟢 低 | 可选事项，可灵活调整 |

### 2.2 冲突检测

生成计划后，检测与已有计划的时间冲突。

**冲突类型**：

| 当前计划 | 已有计划 | 冲突类型 | 说明 |
|:---|:---|:---|:---|
| travel | travel / work(出差) | 🔴 硬冲突 | 无法同时在两地 |
| work | work(同一项目) | 🔴 硬冲突 | 同一人无法同时参与两个关键里程碑 |
| study | study(同时段) | 🔴 硬冲突 | 同一时段无法完成两项深度学习 |
| travel | study / fitness | 🟡 软冲突 | 旅途中可保持轻量学习/运动 |
| work | fitness / study | 🟡 软冲突 | 可安排早晚时段 |

**冲突处理**：检测到冲突时，在计划详情末尾标注冲突信息，允许用户继续保存。

### 2.3 保存并展示

1. 保存计划到 `~/.plan-master-data/plans/plan-{uuid}.md`
2. **立即展示计划详情**（卡片块形式）
3. 如有冲突，在末尾标注
4. **渲染小气泡**（配置见附录A.2场景1）

**展示示例**：
```
## 📋 计划已保存：五一法意瑞10日深度游
━━━━━━━━━━━━━━━━━━━━━━━━
🗺️ 旅游计划 | 🟢 进行中
🎯 核心目标：打卡罗马、巴黎、卢塞恩
📅 时间周期：2026.05.01 - 05.10（10天）

### 🎯 任务清单（9项）
| 事项 | 截止 | 优先级 |
| 预订往返机票 | 3/25 | 🔴 高 |
...

### ⚠️ 冲突提示
🔴 与「Q2项目冲刺」时间重叠（5月5日-10日），请留意安排。
━━━━━━━━━━━━━━━━━━━━━━━━
```

## 步骤3：计划管理

### 3.1 查看计划列表

**触发话术**：「我有哪些计划？」「查看我的计划」

1. 扫描 `~/.plan-master-data/plans/` 目录
2. 展示计划列表
3. **渲染小气泡**（配置见附录A.2场景5）

**展示格式**（紧凑单行）：
```
您当前有 3 个计划：

🗺️ 五一法意瑞游 | 🟢 进行中 | 更新于3/24
💪 Q2健身计划 | 🟢 进行中 | 更新于3/20
📚 雅思备考计划 | 🟢 进行中 | 更新于3/15
```

### 3.2 查看计划详情

**触发话术**：「查看五一旅游计划」「显示 plan-xxx 详情」

1. 读取对应 Markdown 文件
2. **展示完整计划详情**
3. **渲染小气泡**（配置见附录A.2场景6）

### 3.3 修改计划

**触发话术**：「把酒店预订改成4月5日」「添加一个事项：购买旅行保险」

1. 修改 Markdown 文件内容
2. 更新 `updatedAt` 时间
3. **展示修改后的计划详情**
4. **渲染小气泡**（配置见附录A.2场景2）

**回复示例**：
```
✅ 已修改：酒店预订时间改为4月5日

以下是更新后的计划：
[展示完整计划详情]

有需要调整的地方直接告诉我就行～
```

### 3.4 删除计划

**触发话术**：「删除五一旅游计划」

1. 确认删除意图
2. 执行删除
3. 告知用户结果
4. **渲染小气泡**（配置见附录A.2场景3）

### 3.5 删除事项

**触发话术**：「删除去卢浮宫这个项目」

1. 从计划中删除指定事项
2. 更新 Markdown 文件
3. **展示修改后的计划详情**
4. **渲染小气泡**（配置见附录A.2场景4）

### 3.6 生成日程提醒

**触发话术**：「生成日程」「为这个计划创建日程」

> ⚠️ **必须用户主动发起，禁止自动创建！**

1. 筛选 `priority: high` 的高优先级事项
2. 调用 schedule-manager SKILL 创建日程
3. 告知用户已创建的日程
4. **渲染小气泡**（配置见附录A.2场景7）

**日程数量约束**：每个事项单独生成一个日程，禁止合并。

**输出示例**：
```
⏰ 已为你创建了 3 个日程提醒：

| 事项 | 提醒时间 | 重要性 |
|:---|:---|:---:|
| 📌 预订往返机票 | 3月25日 20:00 | 🔴 高 |
| 📌 预订罗马酒店 | 4月1日 20:00 | 🔴 高 |
| 📌 预约梵蒂冈门票 | 4月15日 20:00 | 🟡 中 |

到时间会提醒你，不用担心忘记～
```

### 3.7 生成HTML报告

**触发话术**：「生成报告」「生成计划报告」

1. 读取计划 Markdown 文件
2. 安全检查（过滤敏感内容）
3. 冲突检测并标注
4. 敏感信息脱敏
5. 生成 HTML（遵循 `references/html-standards.md`）
6. 使用 mcporter 调用 `cloud-alipay` 这个 MCP 的 `html_deploy` 工具部署，禁止使用其他部署工具
7. 返回公网链接
8. **渲染小气泡**（配置见附录A.2场景8）

> ⚠️ **强制要求：必须对HTML内容进行转义！**
>
> HTML中包含大量双引号（如 `class="main"`、`href="..."`），直接传入会导致JSON解析失败（报错 `Unterminated string`）。
>
> **必须使用 jq 进行转义**：
> ```bash
> # 正确做法：用 jq -Rs 自动转义HTML中的特殊字符
> HTML_CONTENT=$(cat report.html | jq -Rs)
> mcporter call cloud-alipay.html_deploy --args "{\"html_list\": [$HTML_CONTENT]}"
> ```
>
> **错误做法**（会导致部署失败）：
> ```bash
> ❌ mcporter call cloud-alipay.html_deploy(html_list: ["<html><body class="main">...</body></html>"])
> #                                                              ↑ 未转义的双引号会破坏JSON结构
> ```

**输出示例**：
```
📄 报告已生成！

🔗 报告链接：https://xxx.cloud.alipay.com/plan-xxx.html

你可以分享给朋友或随时查看完整计划。
```
