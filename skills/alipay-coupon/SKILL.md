# 支付宝卡包优惠券查询与领取

查询用户支付宝卡包中的优惠券，同时显示我的券和推荐好券，支持领取操作。

## 适用场景

- 用户想查看自己有哪些优惠券（我的券 + 推荐好券一起展示）
- 用户想领取某张优惠券

## 前置条件

## 命令参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--action` | 操作类型：`query-coupon`（查询）、`get-coupon`（领取） | `query-coupon` |
| `--campId` | 领券活动 ID（领取时必填，从推荐好券结果获取） | |
| `--prizeId` | 领券奖品 ID（领取时必填，从推荐好券结果获取） | |

## 查询优惠券

```bash
node scripts/alipay-coupon.js --action query-coupon
```

输出按商户分组的表格，同时包含我的券（含去使用链接）和推荐好券（含领取参数）。默认展开所有商家下的全部券。

## 领取优惠券

从推荐好券结果中获取 campId 和 prizeId，执行领券：

```bash
node scripts/alipay-coupon.js --action get-coupon --campId xxx --prizeId xxx --scene xxx
```

## 输出格式（强制，不可违反）

> **⚠️ 硬性约束（优先级高于人设和语气要求）：**
> 1. **必须**将脚本输出的 markdown 表格**原样**展示给用户，包括表头、分隔线、每一行数据
> 2. **禁止**省略表格、禁止用散文/列表/摘要等任何非表格形式替代
> 3. **禁止**将表格数据改写为"品牌名 - 金额"这类文字描述
> 4. 推荐文案放在表格**之后**，不能替代表格
> 5. 违反以上任何一条 = 输出错误

```
## 我的券（共 N 张）

### 品牌名
| 优惠 | 券名 | 适用范围 | 有效期 | 操作 |
|------|------|----------|--------|------|
| 5元 满10元可用 | 优惠券名 | 全场可用 | 今天23:59过期 | [去使用](alipays://platformapi/startapp?appId=xxx) |

## 推荐好券

### 品牌名
| 优惠 | 券名 | 适用范围 | 领取 |
|------|------|----------|------|
| 5折 | 券名 | 指定商品可用 | [领取](sendchat://帮我领取美团外卖3元券) |
```

查询结果末尾会附带 HTML 注释形式的领券参数映射（用户不可见），格式如：
```
<!-- 美团外卖3元券: node scripts/alipay-coupon.js --action get-coupon --campId xxx --prizeId xxx --scene HOME_PAGE_LESS_VOUCHER -->
```
收到用户"帮我领取xxx"消息时，从注释中找到对应参数执行领券。**必须带上所有参数（特别是 --scene），缺少会导致领券失败。**

> **重要**：
> - 我的券的"操作"列**必须**包含 `[去使用](alipays://...)` 超链接，链接来自查询结果中的跳转地址。输出前逐行检查，缺少链接的行必须补上。
> - 推荐好券的"领取"列使用 `sendchat://` 协议，格式为 `[领取](sendchat://帮我领取xxx)`，其中 xxx 是该券的品牌+优惠描述（如"美团外卖3元券"），点击后会发送消息触发领券流程。
> - **sendchat:// 链接中禁止包含空格**，空格会导致 markdown 链接解析失败、渲染为原始字符串。用紧凑无空格的描述，如 `sendchat://帮我领取美团外卖3元无门槛券`，而不是 `sendchat://帮我领取美团外卖 3 元无门槛券`。
> - **禁止**用反引号包裹 `sendchat://` 超链接，否则链接无法点击。

## 智能推荐决策（查询后必须执行，禁止跳过）

> **这是本技能的核心环节。** 拿到优惠券数据后，禁止直接列举券信息给用户。你必须先收集环境上下文，再生成有温度的个性化推荐。

### 步骤 A：收集环境上下文（必须并行调用以下工具）

1. **查天气**：调用 `weather` 技能，查询用户所在城市未来 1-3 天天气。如果不确定城市，直接问用户。
2. **查账单**：调用 `call-alipay-service` 账单技能，查询用户最近 3 个月消费记录摘要。关注：高频商家、品类偏好（外卖/到店/出行/购物）、日常消费金额区间、消费时段。

以上两个工具调用**必须执行**，不可省略。它们是推荐决策的数据来源。

### 步骤 B：分析每张推荐券（在你的思考中完成）

对每张推荐好券，依次分析：

1. **拆解券场景**：这张券属于什么消费场景？（出行/外卖/咖啡/生鲜/娱乐/餐饮...）
2. **什么情况下用户会急需它？** 例如：
   - 打车券 → 下雨、深夜、赶飞机、极端天气、带行李
   - 外卖券 → 下雨不想出门、加班到饭点、周末犯懒
   - 咖啡券 → 工作日午后犯困、天气闷热
   - 生鲜券 → 周末做饭、节假日聚餐备菜
   - 火锅券 → 降温、朋友聚会、冬季
3. **匹配用户习惯**：账单数据中，用户是否高频消费该品类/商家？金额门槛是否匹配用户日常消费水平？
4. **发现新场景**：如果用户从未消费过该品类，当前的天气/时间/季节是否正好让这张券变得有价值？这是制造惊喜的机会。

### 步骤 C：生成推荐文案

按以下优先级选 **2-3 张重点推荐**，其余简要提及：

| 优先级 | 类型 | 说明 |
|--------|------|------|
| 最高 | 紧急+匹配 | 即将过期（今天/明天）且匹配用户习惯，必须告诉用户 |
| 高 | 高匹配 | 完全吻合消费习惯（高频商家/品类） |
| 中 | 环境驱动 | 当前天气/时间使某券价值飙升（如下雨天的打车券） |
| 低 | 惊喜发现 | 用户没试过但环境正好适合尝新 |

**文案要求（严格遵守）**：

- **禁止**使用"根据您的消费习惯分析"、"建议您优先领取以下优惠券"等客服话术
- **禁止**用编号列表罗列券名（不是在做报告）
- **必须**像朋友在微信里随口说的一样自然
- **必须**为每条推荐给出**具体理由**——因为什么环境因素/习惯所以值得领
- 紧急的券制造合理紧迫感但不像促销，惊喜推荐自然引导尝新不强推

✅ 正确示范：
> 明天杭州有中雨，正好有张滴滴 5 折打车券后天就过期了，下雨天打车刚好用上，建议先领了备着。看你最近外卖点得挺多的，饿了么这张满 20 减 8 挺适合你，午饭又能省一笔。对了，瑞幸有个买一送一的活动券，你好像没怎么试过，最近加班可以下午来杯提提神，有效期还有两周不急。

❌ 错误示范：
> 根据您的消费习惯分析，您高频使用外卖服务。结合当前天气预报，建议您优先领取以下优惠券：1. 饿了么满20减8 2. 滴滴打车5折券

### 步骤 D：过期提醒（可选）

如有高价值券 3 天内过期，主动问用户要不要设使用提醒（调用 `alipay-cron` 技能）。例如："这张星巴克买一送一明天就过期了，要不要我明天午休前提醒你用？"

## 注意事项

- 跳转链接为 `alipays://` 协议，在支付宝客户端内可直接打开
- 领券需要用户已登录支付宝
